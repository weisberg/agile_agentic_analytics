# Research Notes: 30-Day Experiment Duration Analysis

## Source 1: AB Tasty - How Long Should You Run an A/B Test?
URL: https://www.abtasty.com/blog/how-long-run-ab-test/

### Key Findings:

**Recommended Duration:**
- Average recommended A/B testing time is 2 weeks (14 days)
- Should identify key factors relevant to conversion goals
- Duration depends on multiple criteria

**Critical Factors for Duration:**
1. Statistical confidence level
2. Sample size
3. Representativeness of sample
4. Test period and device being tested

**Business Cycles:**
- Users don't convert immediately - may take 1-3 weeks between exposure and conversion
- If purchasing cycle is 3 weeks and test runs only 1 week, sample is not representative
- **Recommendation: Test over at least one business cycle, ideally two**

**Traffic Sources:**
- Sample must incorporate all traffic sources (email, sponsored links, social networks)
- No single source should be over-represented
- Email campaigns can skew results if they're low traffic but high revenue
- Sales/promotional periods attract atypical users - differences less marked when re-tested outside these periods

**Solutions for Representativeness:**
1. Extend test duration beyond necessary minimum to get closer to normal user distribution
2. Target tests to include only specific population groups (e.g., exclude email campaign users, target only new visitors)

**Key Insight:**
- Difficult to ensure representative samples due to little control over which users participate
- Duration extension is a primary method to overcome sampling bias

## Source 2: Invespcro - How Long Should You Run an A/B Test For?
URL: https://www.invespcro.com/blog/how-long-should-you-run-an-ab-test-for/

### Key Findings:

**Four-Week Maximum Duration:**
- Maximum period recommended: 4 weeks (28 days)
- Rationale: Limit possibility of data pollution by limiting test time

**Why Not Longer (Even for Long Buying Cycles)?**
- Question raised: If buying cycle is 6-8 weeks, shouldn't test cover full cycle?
- Answer: At any point in time, visitors are at different stages in buying funnel
- Tests don't run in controlled environment
- External and internal factors can pollute testing data
- **Key principle: Limit data pollution by capping duration at 4 weeks**

**Problem with Early Stopping:**
- Majority of A/B tests fail to produce meaningful results
- Many marketers too eager to declare winner
- Even when testing software declares winner, good chance it's a false positive
- Need good handle on A/B testing statistics to ensure data collected and analyzed correctly
- Determining duration is critical component of successful conversion optimization program

**Industry Perspective:**
- Some CROs think no need for statistics knowledge - rely on A/B testing software
- This is problematic approach

## Source 3: Statsig - Sequential Testing: How to Peek at A/B Test Results Without Ruining Validity
URL: https://www.statsig.com/perspectives/sequential-testing-ab-peek

### Key Findings:

**The Peeking Problem:**
- Traditional A/B testing: like pregnancy test - wait full time, read result once
- Every time you look at results, you're essentially rolling the dice again
- Check 20 times instead of once? Inflated false positive rate from 5% to much higher
- Constant checking (peeking) tricks you into thinking you've found winner when you haven't

**Sequential Testing Solution:**
- A/B testing that expects you to peek
- Designed to be watched continuously (like heart monitor)
- Adjusts calculations on the fly to keep error rates in check
- Statistical adjustments raise the bar for significance as you check more often

**How It Works:**
- Early in test: need overwhelming evidence to call winner
- As more data comes in: confidence intervals tighten up, threshold for significance becomes more reasonable
- Platform continuously recalculates what "significant" means based on:
  - How long test has been running
  - How many times data has been analyzed

**Key Benefits:**
- Can act on results as soon as genuinely significant (not just statistically noisy)
- Found serious bug tanking conversion? Know within hours, not weeks
- Discovered killer feature? Ship it before competitors finish testing cycle
- Monitor experiments in real-time without guilt or inflated false positive rates

**Strategic Recommendation:**
- Run sequential tests to catch obvious winners and losers early
- Use fixed-horizon tests for nuanced stuff needing careful analysis

## Source 4: Statsig - CUPED Explained
URL: https://www.statsig.com/blog/cuped

### Key Findings:

**What CUPED Stands For:**
- Controlled-experiment Using Pre-Experiment Data
- Coined by Microsoft in 2013
- One of most powerful algorithmic tools for increasing speed and accuracy of experimentation programs

**The Problem CUPED Solves:**
- Experiments often barely miss statistical significance threshold
- Need more sample size to increase chances of significant result
- Standard error/noise goes down with square root of sample size
- Sample size is expensive resource, proportional to enrollment window
- Waiting for more samples delays informed decision-making
- Doesn't guarantee significant result even when real effect exists
- Even at Facebook/Amazon scale, must wait for experiments to mature
- Example: 0.1% revenue increase at Facebook worth $100M+ per year
- For smaller companies, small effect sizes become infeasible to measure

**CUPED Solution:**
- Decreases standard error for same metric and sample size
- Uses pre-experiment data to explain away some variance in result data
- Core principle: Not all variance in experiment is random
- Many differences in user outcomes based on pre-existing factors unrelated to experiment

**How It Works:**
- Uses pre-experiment data for metric to normalize post-experimental values
- Amount of normalization based on how well pre-experiment data predicts experiment data
- Switches focus from raw values to difference from expected values
- "Explains" noise and variance in experiment metric
- Reduces range/variance used to calculate confidence intervals and p-values

**Bias Correction:**
- Random assignment can create groups with different baseline values
- Pre-existing difference could be statistically significant by chance
- Without correction, might conclude difference even if treatment did nothing
- CUPED adjusts: faster baseline group's metric adjusted downwards relative to slower group
- Can follow up with statistical test to understand if adjusted difference is significant

**Advanced Implementations:**
- More complex versions don't rely on single historical data point for same metric
- Can pull in other information independent of experiment group assignment
- Example: age group as factor - has little to do with experiment but major factor in outcomes
- Including additional factors reduces even more variance

**Key Benefit:**
- Helps understand data with more clarity
- Enables faster experiments with same sample size
- Reduces variance without increasing enrollment time

## Source 5: Statsig - Introducing Surrogate Metrics
URL: https://www.statsig.com/blog/introducing-surrogate-metrics

### Key Findings:

**The Problem:**
- Most important metrics for determining feature success are hard to move and materialize over long-term
- Creates tension: want to move fast with experimentation BUT want to make decisions based on long-term high-level metrics
- Opportunity cost of delaying launch of good features

**Surrogate Metrics Definition:**
- Also called proxy or predictive metrics
- Estimate long-term outcome that's impractical to measure during experiment
- Help move fast while staying oriented toward metrics we care most about
- Offer quicker feedback with greater sensitivity to change
- Example: Don't need to wait a year to know if feature worked

**How They Work:**
- Use machine learning model to predict long-term outcome from short-term signals
- Surrogate metric S predicts true north metric X with error term
- Must be unbiased estimator (error term averages to zero)
- Confidence intervals adjusted to account for both experiment variance AND prediction error (MSE)
- Prevents inflation of false positives

**Best Practices:**

1. Validate predictive model and monitor over time
   - Inputs should be independent of assignment
   - No leakage from post-estimation time period
   - Model should be unbiased estimator
   - Outputs should not exhibit heteroscedasticity
   - Product changes can improve/degrade prediction quality over time

2. Measure non-surrogate metrics in experiments too
   - Suite should include measurements of mechanism expected to move true north metric
   - Example: If true north is annual revenue, also track short-term retention, purchase frequency, revenue per purchase
   - Helps understand specific behaviors being influenced
   - Prevents gaming of surrogate metric through abstraction layer

3. Use holdouts to validate decisions made based on surrogate metrics
   - Validate results of individual experiments
   - Opportunity to measure true north metric that surrogate predicts
   - Ensures not being misled by short-term or noisy signals
   - Evaluates decisions in aggregate and/or over longer time period

**Key Benefit:**
- Fast and lightweight decision-making while maintaining connection to long-term outcomes
- Reduces experiment duration by enabling earlier decisions based on predicted long-term impact

## Source 6: Medium/Oda - Practical Guide to Setting A/B Test Duration in Bayesian Context
URL: https://medium.com/oda-product-tech/a-practical-guide-to-setting-a-b-test-duration-in-a-bayesian-context-6a9d685ac5d7

### Key Findings:

**Fundamental Difference from Frequentist:**
- Setting test duration beforehand is concept in Frequentist statistics but NOT in Bayesian statistics
- Frequentist approach offers clear methodology to decide test duration
- Some teams incorrectly apply Frequentist duration methods in Bayesian world
- Lack of information on how to decide A/B test duration within Bayesian framework

**Seven Guidelines for Bayesian Test Duration:**

1. Don't decide on experiment length up front
   - Experiment length shouldn't be decided up front
   - Amazing feature of Bayesian approach (not scary)
   - Don't use frequentist calculators to determine experiment length

2. Reflect on the problem at hand
   - How frequent are conversions? Frequent conversions = faster reliable conclusions
   - Infrequent conversions = longer data collection period needed
   - Example: "orders per user" runs longer than "products added to cart"
   - Will there be novelty effect? Significant UX changes need more time
   - What's effect size vs current mean and standard deviation? High variation = longer duration
   - Impact on revenue, profitability, retention? Add guardrail metrics

3. Choose risk threshold for metrics
   - Bayesian approach is about uncertainties and probabilities
   - Make best decision under uncertainty and acceptable risk
   - Sometimes conclude earlier than thought (certain early on)
   - Other times run longer and accept some risk
   - Choose desired error tolerance (risk) willing to accept
   - Run until expected loss is below specified tolerance
   - Be active participant in decision making rather than passive recipient of results

4. Choose minimum duration
   - Due to cyclical nature of business (weekly shopping), choose minimum duration capturing full cycle
   - Don't run experiments shorter than week
   - Always run for full cycles to rule out cycle effect

5. Consider whether constrained by maximum duration
   - Healthy exercise not done often enough
   - Consider external deadlines for communicating results
   - Set as experiment's maximum duration
   - Compare acceptable risk to risk provided by experiment on end date
   - Alternative: set time when want to make decision or iterate on treatment
   - Learning from short iteration, tweaking, starting new iteration = good lean product development practice
   - Bigger risks if run for shorter time
   - If have time/can afford cost, add another cycle (won't harm results, metrics stabilize, risk reduced)

6. Decide what to do during first week
   - Define minimum number of conversions per metric before showing results to stakeholders
   - If very negative results or data quality issues early on, kill experiment right away

7. Decide when to stop experiment
   - Can stop at any time and make perfectly valid inferences (Evan Miller)
   - BUT: Bayesian approach NOT immune to peeking problem (nuanced)
   - General rule: wait until risk of choosing treatment is below acceptance threshold
   - If doesn't happen and constrained by maximum duration, stop and collect findings

**Key Insight:**
- Bayesian approach allows active participation in decision-making by looking at overall uplifts, probabilities, risks and making business call
- Contrast with Frequentist: sitting back and taking results at end of experiment

## Source 7: Statsig - Novelty Effects: Everything You Need to Know
URL: https://www.statsig.com/blog/novelty-effects

### Key Findings:

**Definition:**
- Short-term effects in product metrics attributed to introduction of something new
- Something new evokes temporary reaction that wears off as newness wears off
- Response to new feature temporarily deviated from inherent value due to newness
- Keyword: "temporarily" - given enough time, novelty effects wear off

**Impact on Experiment Duration:**
- When study observed effect, look at cumulative effects (delta between treatment and control cumulative throughout duration)
- Drawing conclusions from cumulative effects allows more statistical power
- Fine if expected treatment effects don't change
- NOT fine with novelty effects - even short-term metrics like CTR behave differently when newness wears off
- Should look at daily view of treatment effects or "days since exposure" view

**Pattern Recognition:**
- Usually see higher daily delta on CTR at beginning, then becomes lower and stays lower
- But other direction can happen too
- Example: Facebook changed from timestamp-based news feed (EdgeRank) to ML-curated feed - users' initial reactions not welcoming

**How to Safely Control for Novelty Effects:**

Eye-balling data has challenges:
1. Apophenia (pattern-matching biases) tricks into conjuring conclusions not there
2. Lack of statistical power

Recommendations:
1. Use "days since exposure" view in experiment results to find suspects of novelty effects
2. Shallow metrics like CTR behave differently between short-term and long-term
3. Throw away first few days of data where results might be driven by novelty effects
4. Make decisions using remaining data where effect converges to equilibrium, long-term value

**When Novelty Effects Matter:**
- Novelty effects are luxury
- For user to feel novelty, needs expectations of experience with product
- Not case for most businesses
- Products where users can detect differences in experience see largest manifestation
- Degree of change matters
- Formula: Novelty ∝ Surprise × Frequency
- Example: McDonald's UI change may not trigger novelty effects, but re-introducing McRibs probably will

**Key Insight:**
- Novelty effects not bad, actually sometimes valuable
- Tempting to solve with statistical tests but generally not advisable
- Downside of ignoring temporary deviations is low
- Should be cautious about sentencing temporary deviation as novelty effects

**Implication for 30-Day Duration:**
- Need sufficient time for novelty effects to wear off
- First few days may not be representative of long-term effect
- Longer durations help reach equilibrium value

## Source 8: Research on Optional Stopping and Multiple Testing

### Key Findings from Search Results:

**Optional Stopping Problem:**
- Practice of peeking at data and deciding whether to continue experiment based on results
- Can cause true hypotheses to be wrongly rejected much more often than intended
- Repeatedly analyzing data during collection and deciding to stop if statistical significance reached
- Inflates false positive rate when using frequentist methods
- Debate: "Optional stopping: no problem for Bayesians" (Rouder 2014) vs "Why optional stopping can be a problem for Bayesians" (counterargument)

**Sequential Stopping Rules:**
- Rule or procedure for determining when experiment should end
- Used in clinical trials and simulation studies
- Must be selected carefully - investigator must abide by stopping rules after first test
- Different from fixed-sample designs where sample size determined in advance

**False Discovery Rate (FDR):**
- Method of conceptualizing rate of Type I errors in null hypothesis testing when conducting multiple comparisons
- Way to identify as many significant features as possible while incurring relatively low proportion of false positives
- Controls error rate among set of tests
- Benjamini & Hochberg (1995) - highly cited foundational work (124,875 citations)

**Implications for Experiment Duration:**
- Fixed duration protects against optional stopping problem in frequentist framework
- Pre-specified duration prevents data-dependent stopping that inflates false positive rate
- Trade-off between flexibility and statistical validity
- Sequential methods require special adjustments to maintain validity
