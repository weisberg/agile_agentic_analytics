## Phase 1: Randomization and Balance Checks

### Source: Statsig - Randomization: The ABC's of A/B Testing
URL: https://www.statsig.com/blog/randomization-the-abcs-of-a-b-testing

**Key Concepts:**

1. **Simple Randomization**
   - Relies on random assignment to balance groups naturally
   - Critical requirement: randomization mechanism must be genuinely random and unaffected by treatment
   - Does NOT guarantee comparability - can lead to imbalances in key attributes
   - Most problematic when certain characteristics strongly influence test outcomes

2. **Seed Randomization**
   - Middle-ground approach between random and controlled
   - Generates multiple random groupings BEFORE experiment begins
   - Uses historical data to evaluate group balance on key factors
   - Process: repeatedly divide users → evaluate balance → apply new seed if imbalanced → repeat until acceptable balance achieved
   - Advantage: Better balance than simple randomization while maintaining randomness

3. **Stratified Randomization (Blocked Randomization)**
   - Most controlled approach
   - Uses historical data BEFOREHAND to deliberately construct comparable groups
   - Process reversed from seed: assess characteristics first → construct groups based on attributes
   - Also called: stratified sampling, block randomization, randomized block design
   
   **Implementation:**
   - Select key characteristics for balancing
   - Group users into strata based on characteristic combinations
   - Example strata: (paying/non-paying) × (long-term/recent) = 4 strata
   - Within each stratum: use block randomization with predetermined block size
   - Generate randomized sequence of group labels (e.g., ABBA, BBAA)
   - Ensures balanced proportions within each stratum

**Critical Insight for Early Diagnostics:**
- Randomization validation is ESSENTIAL in first hours
- Without proper randomization, "no A/B test can be considered valid"
- Must verify randomization mechanism is unaffected by treatment and stable across time

## Phase 1 (continued): Sample Ratio Mismatch (SRM) Detection

### Source: Microsoft Research - Diagnosing Sample Ratio Mismatch in A/B Testing
URL: https://www.microsoft.com/en-us/research/articles/diagnosing-sample-ratio-mismatch-in-a-b-testing/

**Critical Concept: Survivorship Bias in A/B Testing**
- SRM = statistically significant difference between observed user ratio vs configured ratio (e.g., 50/50 split)
- Missing users are rarely random - often those most impacted by the treatment
- Example: MSN carousel test showed decreased engagement due to bot detection filtering out most engaged users
- After resolving SRM, results flipped to show positive impact

**Prevalence of SRM:**
- LinkedIn: ~10% of zoomed-in A/B tests suffer from SRM
- Microsoft: ~6% of all A/B tests have SRM
- This is a widespread, critical quality issue

**SRM Taxonomy - Root Causes by Stage:**

1. **Assignment Stage**
   - Incorrect bucketing of users
   - Faulty User IDs
   - Carry-over effects from previous experiments

2. **Execution Stage**
   - Redirecting users in one variant
   - Variant changing engagement levels
   - Performance differences affecting data collection

3. **Log Processing Stage**
   - Incorrect data joins
   - Data pipeline issues

4. **Analysis Stage**
   - Using biased conditions to segment analysis
   - Misconfigured triggered conditions

5. **Experimenter-Caused**
   - Uneven ramp-up of experiment variants
   - Users self-assigning into variants

**SRM Detection Method:**
- Must test BEFORE analyzing experiment results
- Use chi-square statistic (not just ratio comparison)
- Need sample size information, not just ratio
- Microsoft threshold: p-value < 0.0005 (conservative to reduce false positives)
- Tests with SRM typically have much lower p-values than threshold

**Diagnostic Steps (Differential Diagnosis):**

1. **Segment Analysis**
   - Check if SRM is localized to particular segments/cohorts
   - Impacted segments will have extremely low p-values
   - Example: SRM might affect only specific browser types
   - Visualization: red cells = SRM detected, gray = clean, cell size = segment size

2. **Triggering Condition Analysis**
   - For triggered analyses (subset of assigned population)
   - Examine boolean condition used to filter logs
   - Common issue: condition not valid in all variants (lack of counterfactual logging)
   - Diagnostic test: check if untriggered population has no SRM
   - If untriggered is clean but triggered has SRM → misconfigured condition

3. **Additional Diagnostic Signals**
   - **Significance of SRM**: Very low p-value = very severe root cause
   - **Directionality**: More users in treatment often indicates increased engagement or better performance
   - **Count of SRMs**: Multiple simultaneous SRMs could indicate data pipeline issue
   - **Temporal analysis**: When did SRM root cause occur?

**Key Principle for Early Diagnostics:**
"Don't trust the results of A/B tests with an SRM until you diagnose the root cause."
- SRM check must be integrated into every mature A/B testing platform
- Must be performed on underlying data (user counts) before ship-decision analysis
- Acts as end-to-end method for detecting severe quality issues

## Phase 1 (continued): Leading vs Lagging Indicators

### Source: Statsig - Leading vs. lagging indicators: Experimenting to find the best predictors
URL: https://www.statsig.com/perspectives/leading-vs-lagging-indicators

**Core Definitions:**

**Leading Indicators:**
- Predictive metrics that guide future actions and strategies
- Focus on user behavior data
- Help anticipate trends and outcomes
- Enable proactive decision-making
- Provide insights into potential future performance
- Guide immediate actions and adjustments

**Lagging Indicators:**
- Retrospective measures of past performance and outcomes
- Confirm results of previous strategies and actions
- Often relate to revenue and success measurement
- Provide clear picture of what has already happened
- Help evaluate effectiveness of strategies over time

**Key Difference:**
- Leading indicators FORECAST future trends
- Lagging indicators VALIDATE past results
- Both types needed for informed decisions and strategy alignment

**Practical Examples:**

Leading Indicators:
- Session duration and frequency (reflect engagement and potential retention)
- Number of users actively engaging with platform
- Conversion rates from free to paid users (signal potential revenue growth)
- Marketing-generated pipeline volume
- User engagement metrics
- Content quality measures

Lagging Indicators:
- Revenue metrics: ARR (Annual Recurring Revenue), MRR (Monthly Recurring Revenue)
- Customer churn rate (reflects retention effectiveness)
- Sales cycle length (measures conversion efficiency)
- Marketing-generated revenue
- Daily Active Users (DAU) - long-run measure

**Critical Insight on Correlation:**
- Increase in marketing-generated pipeline volume (leading) should correlate with increase in marketing-generated revenue (lagging)
- Directly measuring content quality can improve estimates when predicting long-run effect on DAU from short-run engagement
- Must establish clear connections between leading and lagging metrics

**Challenges in Measurement:**
- User behavior is complex
- Data can be noisy
- Accurately measuring leading indicators is challenging
- Need analytical tools to track and analyze experimental data effectively
- Must continuously experiment and refine leading indicators

**Integration for Decision-Making:**
- Combine both types for complete performance picture
- Link user engagement metrics (leading) directly to revenue growth (lagging)
- Regularly monitor both types to identify trends
- Use leading indicators to fine-tune strategies in real-time
- Use lagging indicators to assess overall success over time
- Continuously analyze relationships between leading and lagging indicators

**Experimentation Role:**
- Critical for identifying which leading indicators best predict desired outcomes
- Run controlled experiments to isolate impact of specific metrics on business goals
- Continuously refine understanding through experimentation
- Share experimental findings for community feedback

## Phase 1 (continued): Guardrail Metrics System

### Source: Airbnb Engineering - Designing Experimentation Guardrails
URL: https://medium.com/airbnb-engineering/designing-experimentation-guardrails-ed6a976ec669

**Purpose:**
- Protect key metrics by identifying potentially negative impacts PRIOR to launch
- Prevent experiments from harming most important metrics while allowing experimentation at scale
- Experiments that trigger guardrails go through escalation process with stakeholder discussion

**Three Categories of Guardrail Metrics:**

1. **Key business/financial metrics** - represent overall company performance (e.g., revenue)
2. **User experience metrics** - capture product feel (e.g., bounce rate, page load speed)
3. **Strategic priority metrics** - focus on strategic importance areas (e.g., specific product adoption)

**Critical Trade-off:**
- More guardrail metrics ≠ better protection
- Balance between: metrics protected, thoroughness of protection, friction in development process
- Example: 50 metrics with 0.05 significance → 92% false alert rate in AA test

**Three Guardrails Framework:**

**1. Impact Guardrail**
- Escalates if: percent_change < -t (where t = escalation threshold)
- Example: if t = 0.5%, impact more negative than -0.5% triggers escalation
- Protects against large negative effects REGARDLESS of statistical significance
- Coverage-adjusted: t varies with global coverage (% of visitors assigned to experiment)

**2. Power Guardrail**
- Ensures experiment exposed to enough users
- Requires: StdError < 0.8 * t
- Impacts experiment runtime
- Ensures Impact Guardrail has reasonably good power and false positive rate (FPR)
- Longer runtime → smaller FPR and larger power to detect impact
- More metrics → larger experiment-level FPR at any runtime

**3. Stat Sig Negative Guardrail**
- Escalates if: percent_change < 0 AND p-value < 0.05
- Extra safeguard for TOP metrics where even small negative impact should be surfaced
- Not applied to all metrics - only where small degradation is material
- Example: 0.1% revenue decrease at scale = millions of dollars

**Global Coverage Adjustment:**
- Global coverage = % of visitors assigned to experiment
- Escalation Parameter T set for each metric (percent change triggering escalation at 100% coverage)
- Threshold t varies with coverage: t = T / sqrt(coverage)
- Lower coverage → higher threshold on percent change (makes Power Guardrail pass rate similar)
- Lower coverage → lower threshold on global impact (tougher on low-coverage experiments)

**Automatic Approval Refinements:**

1. For metrics where statistical significance is easy but only large changes are material:
   - Ignore Stat Sig Guardrail
   - Apply only Power and Impact Guardrails

2. For experiments not yet passing Power Guardrail but with positive point estimates:
   - Allow pass without escalation if noninferiority test passes
   - Lower bound of confidence interval must satisfy: LowerBound > -0.8 * t

**Implementation Results at Airbnb:**
- Flags ~25 experiments per month for escalation/review
- 80% eventually launch after stakeholder discussion and additional analysis
- 20% (5 per month) stopped before launch
- Balance between safeguarding metrics and maintaining nimble development

**Key Configuration Decisions:**

1. **Setting Parameter T:**
   - As much art as science
   - Set T as LARGER of:
     - "What impact is worth escalating for?"
     - "What impact is feasible to detect?"
   - Tighter T catches smaller impacts but makes Power Guardrail harder to pass

2. **Number of Guardrail Metrics:**
   - More metrics → higher overall escalation rate and false positive rate
   - Escalations slow iteration and launch speed
   - Must balance protection vs moving quickly
   - Use historical data to evaluate different configurations

**Critical Insight for Early Diagnostics:**
- Guardrails must be checked BEFORE launch decisions
- System provides visibility into most important metrics
- Enables transparent discussion of trade-offs between teams
- Prevents post-launch discovery of significant harm requiring rollback

## Phase 2: AA Testing for Validation

### Source: Statsig - What are A/A tests? Validating experiment setup
URL: https://www.statsig.com/perspectives/aa-tests-validating-setup

**Core Purpose:**
- Validate accuracy and reliability of experimentation setup
- Test two IDENTICAL versions against each other
- Ensure testing tools and configurations are working correctly
- Focus on verifying integrity of testing infrastructure (vs A/B tests which identify improvements)

**Expected Outcome:**
- Ideally should show NO significant differences between two identical versions
- Any differences indicate problems with setup, not actual treatment effects
- Ensures differences in A/B tests are due to changes made, not errors in data collection or tool configuration

**When to Run AA Tests:**

1. Regularly as best practice
2. After implementing new tools
3. After making significant changes to testing setup
4. When trying out new experimentation platform
5. To establish reliable baseline before A/B testing

**Key Benefits:**

1. **Data Collection Validation**
   - Confirms tools collecting and recording data correctly without hiccups
   - Validates experimentation platform working as expected
   - Catches technical glitches early

2. **Baseline Metrics Establishment**
   - Serves as reference point for setting realistic goals in later experiments
   - Enables informed decisions based on solid, accurate data

3. **Sample Ratio Mismatch Detection**
   - SRM = traffic not evenly split between variants
   - Skews results and introduces bias
   - Catching SRM early prevents wrong conclusions

4. **Platform Validation**
   - Especially important when trying new testing tools
   - Validates experimentation platform functionality
   - Gives confidence in future A/B test results

**Running Effective AA Tests:**

**Setup:**
- Randomly divide users into equal groups
- Ensure IDENTICAL conditions (no differences between groups)
- Sets up reliable baseline

**Execution:**
- Let test run long enough to achieve statistical significance
- RESIST urge to peek at data before test completes
- Checking too early leads to false positives

**Analysis:**
- Look for discrepancies or anomalies
- Watch for sample ratio mismatches (SRM)
- If significant differences found → dig deeper to find why
- Could indicate: platform problems, data collection errors, technical glitches

**Best Practices:**

1. Run AA tests regularly, especially after tool tweaks
2. AVOID peeking before test finishes (leads to false positives and incorrect conclusions)
3. Let AA test run full course to collect enough data for meaningful analysis
4. Balance reliability and efficiency in planning

**Limitations and Trade-offs:**

1. **Resource Costs:**
   - Take up time and resources
   - Could be used for revenue-generating A/B tests instead

2. **Opportunity Costs:**
   - Time spent on AA tests instead of potentially impactful A/B tests
   - Must carefully plan and prioritize AA testing efforts

3. **Time-Consuming:**
   - Pulls resources away from experiments that could drive significant improvements
   - Must consider trade-offs and allocate resources wisely

**Critical Insight for Early Diagnostics:**
- AA tests are crucial FIRST STEP before running actual experiments
- Validate setup integrity before investing in A/B testing
- Prevent costly mistakes and false conclusions
- Ensure trustworthy data foundation for all future experiments

## Phase 2 (continued): Statistical Power in A/B Testing

### Source: Statsig - Understanding statistical power in A/B testing
URL: https://www.statsig.com/perspectives/understanding-statistical-power-ab-testing

**Core Definition:**
- Statistical power = test's ability to detect a real effect when one truly exists
- Reflects likelihood that A/B test will reveal genuine difference between variants
- Probability of correctly rejecting null hypothesis when true difference exists

**Why It Matters:**

1. **Avoids Type II Errors (False Negatives)**
   - False negatives = test fails to identify significant changes that could boost conversions/revenue
   - Missing valuable insights due to insufficient power
   - Real differences between variants go undetected
   - Potential gains slip through cracks

2. **Increases Confidence in Results**
   - High power ensures observed differences aren't just due to chance
   - Enables implementation of changes that drive meaningful improvements
   - Leads to better business decisions
   - Properly powered tests yield reliable, actionable insights

3. **Impacts Efficiency and Reliability**
   - Well-powered tests enable informed decisions based on accurate results
   - More effective optimizations and better business outcomes
   - Underpowered tests waste resources and slow progress
   - Fail to identify meaningful changes

**Standard Power Level:**
- Usually 80% or higher
- Means test will detect significant changes when they truly exist
- Reduces risk of missing real effects

**Key Factors Influencing Statistical Power:**

**1. Sample Size**
- Most straightforward factor
- Larger sample sizes → increased likelihood of detecting significant differences
- Can detect even small effect sizes with large samples
- Not always practical due to resource constraints or low traffic

**2. Minimum Detectable Effect (MDE)**
- Smallest difference between variants test can reliably detect
- Smaller MDE → larger sample size needed to maintain desired power
- Must balance what you want to detect vs what's feasible

**3. Significance Level (Alpha)**
- Lowering alpha (e.g., 0.05 to 0.01) reduces false positives
- But requires more data to maintain power
- Trade-off between Type I and Type II errors

**4. Base Conversion Rate**
- Low base conversion rate → larger sample required to spot same effect size
- Affects variance in measurements
- Critical for sample size calculations

**Calculating and Optimizing Power:**

**Pre-Test Calculations:**
- Use sample size calculators
- Input: effect size, significance level, base conversion rate
- Output: number of users per variant needed
- Plan test duration based on traffic levels

**Optimization Strategies for Limited Samples:**
1. Increase MDE (detect larger effects)
2. Extend test duration
3. Leverage historical data
4. Adjust significance level (carefully)

**Post-Test Power Analysis:**
- Important when results aren't significant
- Low power might mean you missed real effect
- Consider power level before concluding no difference exists
- Don't assume non-significant = no effect if power was low

**Critical Warnings:**

1. **Don't Set Sample Size Too Large Without Reason**
   - Could detect differences that aren't practically meaningful
   - Statistical significance ≠ practical significance
   - Resource waste on detecting trivial effects

2. **Balance Required:**
   - Must balance: detection sensitivity, resource constraints, practical significance
   - Consider business impact of effect sizes you're trying to detect

**Implications for Early Diagnostics (24-48 hours):**
- Early in experiment, power is LOW due to small accumulated sample
- Cannot reliably detect effects in first 24-48 hours unless:
  - Effect size is very large
  - Traffic is extremely high
  - Using sequential testing methods
- Early "significance" often spurious due to low power
- Must resist temptation to stop early based on apparent significance

**Key Principle:**
"Statistical power isn't just a numbers game; it directly impacts how efficient and reliable your A/B testing process is."

## Phase 2 (continued): The Peeking Problem and Sequential Testing

### Source: Statsig - Sequential testing: How to peek at A/B test results without ruining validity
URL: https://www.statsig.com/perspectives/sequential-testing-ab-peek

**The Peeking Problem Defined:**
- Traditional A/B testing assumes: pick sample size → run to completion → check ONCE at end
- Reality: practitioners check results continuously as data streams in
- Every time you look at results = essentially rolling dice again
- Check 20 times instead of once → false positive rate inflates from 5% to potentially 40%+ 
- "Significant" result might just be statistical noise
- Traditional fixed-horizon tests not built for always-on, continuously monitored world

**Why Peeking Matters:**
- Temptation to peek is irresistible when seeing real-time data
- Quick decisions often matter more than perfect statistics
- But peeking without proper framework invalidates statistical guarantees
- Can lead to false conclusions and poor business decisions

**Sequential Testing as Solution:**

**Core Concept:**
- A/B testing that EXPECTS you to peek
- Designed to be watched continuously (like heart monitor vs pregnancy test)
- Adjusts calculations on-the-fly to keep error rates in check
- Acknowledges reality instead of pretending practitioners won't peek

**How It Works:**
- Statistical adjustments raise bar for significance as you check more often
- EARLY in test: need overwhelming evidence to call winner
- As more data accumulates: confidence intervals tighten, threshold becomes more reasonable
- Platform continuously recalculates what "significant" means based on:
  - How long test has been running
  - How many times data has been analyzed

**Key Advantage:**
- Can act on results as soon as genuinely significant (not just noisy)
- Serious bug tanking conversion? Know within HOURS, not weeks
- Killer feature crushing it? Ship before competitors finish testing
- Speed without sacrificing statistical validity

**Implementation Requirements:**

**1. Tuning Parameters (control conservatism vs aggressiveness):**
- Catch big effects quickly → tune for speed
- Worried about false positives → dial up conservatism
- Need precise effect size estimates → plan to run longer

**2. Stopping Rules (conditions that end test):**
- Stop when hit significance (positive or negative)
- Stop after maximum time period
- Stop when confidence interval narrow enough

**3. Dynamic Confidence Intervals:**
- Early: wide intervals (like training wheels for statistical conclusions)
- Over time: tighten up as evidence becomes clearer

**Interpretation Best Practices:**
- Look at TRENDS, not moments
- Spike at specific time? Probably noise
- Consistent lift over multiple days? Real signal
- Early winners often become losers by day 7

**Practical Advice:**

**DO:**
- Start with hypothesis, not fishing expedition
- Define key metric upfront (don't hunt for significance)
- Account for weekly patterns (Monday ≠ Friday)
- Run important tests to completion even if hit early significance
- Document decision criteria BEFORE starting
- Watch out for novelty effects (early spike might be exploration)
- Consider traffic patterns (B2B vs consumer cycles differ)

**Use Sequential Testing For:**
- Catching bugs and regressions fast
- Ship/no-ship decisions on single metrics
- High-traffic experiments (data accumulates quickly)
- Situations where speed > precision
- Identifying regressions in hours vs weeks

**Use Traditional Fixed-Horizon Tests For:**
- Precise effect size estimates needed
- Multiple metrics matter equally
- High cost of wrong decision
- Testing subtle UX changes
- Nuanced analysis required

**Trade-offs:**
- Trading some statistical power for faster decisions
- If don't find significance early, harder to detect smaller effects later
- Like binoculars great for spotting elephants but might miss mice
- Speed vs precision balance

**Critical Warnings:**
- Don't peek at segments unless adjusted for multiple comparisons
- Data peeking isn't evil - peeking WITHOUT proper framework is the problem
- Easy to move goalposts when watching results (hence pre-documentation)

**Implications for 24-48 Hour Diagnostics:**
- Sequential testing enables VALID early monitoring
- Can identify major issues (bugs, regressions) within first 24-48 hours
- But still need overwhelming evidence for early stopping
- Early signals must be interpreted with appropriate statistical adjustments
- Confidence intervals very wide in first 24-48 hours
- Trends over days more reliable than single-day spikes

**Key Principle:**
"Sequential testing isn't a magic bullet, but it's a powerful tool for the modern experimenter's toolkit. It lets you have your cake and eat it too - continuous monitoring without the statistical hangover."

## Phase 3: Novelty Effects and Temporal Patterns

### Source: Statsig - Novelty effects: Everything you need to know
URL: https://www.statsig.com/blog/novelty-effects

**Core Definition:**
- Novelty effects = short-term effects in product metrics attributed to introduction of something new
- Something new evokes temporary reaction that wears off as newness wears off
- Response to new feature temporarily deviated from inherent value due to newness
- Keyword: "TEMPORARILY"

**Critical Characteristics:**

1. **Temporary Nature:**
   - Given enough time, novelty effects WILL wear off
   - Short-term metrics like CTR behave differently when newness wears off
   - Effects converge to equilibrium, long-term value

2. **Directional Variability:**
   - Usually: higher daily delta at beginning → becomes lower and stays lower
   - But opposite can happen: initial negative reaction → improves over time
   - Example: Facebook's EdgeRank to ML-curated feed - users' initial reactions not welcoming

3. **Not Universal:**
   - Not all products have novelty effects
   - Exist mostly in HIGH-FREQUENCY products
   - Products used frequently more susceptible to novelty reactions

**Why Statistical Tests Don't Solve Novelty:**
- Tempting to use statistical tests but NOT advisable
- Problem: expected treatment effects DO change over time with novelty
- Cumulative effects analysis assumes stable treatment effect (violated by novelty)
- Drawing conclusions from cumulative effects gives more statistical power BUT misleading with novelty

**Necessary Condition for Novelty Effect:**
- Short-term effects are DIFFERENT from long-term effects
- Effects WEAR OFF over time
- Shallow metrics (like CTR) behave differently between short-term and long-term

**Detection Method:**

**Use "Days Since Exposure" View:**
- Examine daily view of treatment effects (not just cumulative)
- Look for pattern: higher daily delta at beginning → lower and stable later
- Shallow metrics show different behavior between short-term and long-term

**Recommended Approach:**

1. **Identify Suspects:**
   - Use "days since exposure" view in experiment results
   - Look for shallow metrics (CTR, engagement) behaving differently short vs long-term

2. **Discard Early Data:**
   - Throw away first few days of data where results driven by novelty
   - Typical discard period: first 3-7 days depending on product

3. **Make Decisions on Remaining Data:**
   - Use data where effect converges to equilibrium, long-term value
   - Ensures product decisions based on sustained impacts, not temporary reactions

**Implications for 24-48 Hour Diagnostics:**

**CRITICAL WARNING:**
- First 24-48 hours MOST susceptible to novelty effects
- Early engagement spikes may NOT represent true long-term effect
- Shallow metrics (CTR, clicks, initial engagement) especially unreliable early
- Cannot distinguish between:
  - Genuine improvement
  - Temporary novelty-driven spike
  - User exploration/curiosity

**What IS Reliable in 24-48 Hours:**
- Technical diagnostics (SRM, randomization checks)
- Exposure delivery validation
- Invariant metrics (should not change)
- Catastrophic failures (major bugs, crashes)

**What is NOT Reliable in 24-48 Hours:**
- Engagement metrics (CTR, time spent, interactions)
- Shallow behavioral metrics
- User preference signals
- Conversion metrics (unless very high volume)

**Key Principle:**
"Ignoring the temporary nature of novelty effects may lead to incorrect product decisions, and worse, bad culture."

**Best Practice:**
- Monitor early signals for catastrophic issues
- DO NOT make ship decisions based on 24-48 hour engagement data
- Wait for effects to stabilize (typically 7+ days)
- Distinguish learning signals (early diagnostics) from decision signals (stabilized effects)

## Phase 3 (continued): CUPED Variance Reduction

### Source: Statsig - CUPED Explained
URL: https://www.statsig.com/blog/cuped

**Full Name:** Controlled-experiment Using Pre-Experiment Data
**Origin:** Microsoft, 2013

**Core Problem CUPED Solves:**
- Experiments often show results "only barely" outside statistical significance range
- Need more sample size to increase chances of significant result
- Standard error/noise goes down with SQUARE ROOT of sample size
- Sample size expensive - proportional to enrollment window
- Waiting delays informed decisions and doesn't guarantee significance
- Even at massive scale (Facebook, Amazon), must wait for experiments to mature
- Small effect sizes can be infeasible to measure for smaller companies

**Economic Impact Example:**
- 0.1% increase to revenue at Facebook = $100+ million per year
- Small effects extremely valuable but hard to detect

**CUPED Solution:**
- Decrease standard error for SAME metric and sample size
- Uses pre-experiment data to explain away variance in result data
- One of most powerful algorithmic tools for increasing speed and accuracy of experimentation

**Core Statistical Concept:**

**Guiding Principle:**
"Not all variance in an experiment is random. A lot of differences in user outcomes based on pre-existing factors that have nothing to do with experiment."

**Mechanism:**
1. Use pre-experiment data for metric to normalize post-experimental values
2. Switch from caring about raw metric to caring about DIFFERENCE from what we'd expect
3. "Explain" some of noise and variance in experiment metric
4. Reduce range/variance used to calculate confidence intervals and p-values

**How Much to Normalize:**
- Based on how well pre-experiment data predicts experiment data
- Uses correlation between pre and post data
- Optimal adjustment factor θ = cov(Y,X) / var(X)
- Variance reduction = var(Y) × (1 - ρ²) where ρ = correlation

**Bias Correction Capability:**

**Problem:**
- Random assignment can create groups with different baseline values
- Pre-existing difference could be statistically significant by chance
- Even if treatment does nothing, might conclude there's difference

**CUPED Correction:**
- If one group has faster average baseline, their experiment results also faster
- CUPED adjusts faster group's metric DOWNWARDS relative to slower group
- Corrects for pre-existing imbalances
- Can reveal true treatment effects masked by baseline differences

**Implementation Variants:**

**1. Parametric CUPED:**
- Uses regression-based adjustment
- Continuous adjustment based on pre-experiment values
- T-test mathematically equivalent to regression with experiment group as independent variable

**2. Stratified/Bucketed CUPED:**
- Non-parametric approach
- Split users into groups based on pre-experiment values
- Measure metrics relative to average metric value of that group
- Example: bucket users by baseline behavior ranges

**3. Multi-Covariate CUPED:**
- Don't rely on single historical data point
- Pull in other information independent of experiment group
- Can include demographic factors, usage patterns, etc.
- Reduces even more variance by accounting for multiple factors

**Critical Requirements:**

**Independence Condition:**
- Covariates must be independent of experiment group assignment
- Cannot use post-randomization data
- Must be measured BEFORE experiment starts
- Violating independence introduces bias

**Typical Pre-Experiment Window:**
- Commonly use 7-14 days of pre-experiment data
- Amplitude Experiment: uses 14 days by default
- Balance between:
  - Longer window = more stable baseline
  - Too long = less predictive of current behavior

**Relationship to Regression:**
- T-test for metric = regression where dependent variable is metric, independent variable is experiment group
- CUPED adds pre-experiment data as covariate
- Mathematically equivalent to OLS regression with covariates

**Variance Reduction Benefits:**

**Typical Improvements:**
- Can reduce variance by 30-50% in many cases
- Some implementations report up to 85% variance reduction
- Depends on correlation between pre and post metrics
- Higher correlation = greater variance reduction

**Speed Benefits:**
- Can reach statistical significance FASTER
- Effectively "doubles" sample size in terms of statistical power
- Reduces experiment duration needed
- Enables detection of smaller effects

**Implications for 24-48 Hour Diagnostics:**

**Advantages:**
- CUPED can help detect effects earlier by reducing noise
- Particularly valuable when pre-experiment data highly correlated with outcome
- Can improve early signal detection for metrics with stable baselines

**Limitations:**
- Still requires sufficient sample accumulation
- Cannot overcome fundamental power limitations
- Most effective when:
  - High correlation between pre and post metrics
  - Stable user behavior patterns
  - Sufficient pre-experiment data available

**Not a Magic Solution:**
- Doesn't eliminate need for adequate sample size
- Cannot detect effects that don't exist
- Still subject to novelty effects in early days
- Variance reduction helps but doesn't change underlying effect size

**Best Use Cases:**
- Metrics with strong temporal correlation (e.g., revenue, engagement)
- Users with established behavioral patterns
- Experiments where baseline imbalances suspected
- High-variance metrics that need sensitivity boost

**Key Principle:**
"CUPED is frequently cited as—and used as—one of the most powerful algorithmic tools for increasing the speed and accuracy of experimentation programs."

## Phase 4: Early Decision-Making Framework

### Source: Statsig - An overview of making early decisions on experiments
URL: https://www.statsig.com/blog/making-early-decisions-on-experiments

**Core Problem:**
"Something I see customers/practitioners run into all too often is basing their conclusions on tests too soon in the process."

**Risk vs. Reward Framework:**

**RISKS of Early Decisions:**

1. **Novelty Effects:**
   - May influence early results
   - Initial response does NOT indicate long-term behavior
   - Examples: tests on new products gone viral over weekend, just started ad campaign
   - Can lead to noisy data interpretation

2. **Noisy Data:**
   - Early data can be noisy
   - May not represent true effect of experiment
   - Leads to incorrect conclusions
   - Higher likelihood of false positives AND false negatives

**REWARDS of Early Decisions:**

1. **Early Insights:**
   - Quick feedback enables faster iteration
   - Faster improvement of features
   - Speed to market advantages

2. **Resource Allocation:**
   - Identify strong positive or negative trend early
   - Decide whether to continue investing resources
   - Prevent wasted effort on clearly failing experiments

**Consequences of Acting Too Soon:**
- Think something works when it doesn't (false positive)
- Kill something that does work (false negative)
- Mislead teams
- Waste time and resources
- Overlook important findings
- Make plans that don't match user wants or business needs

**Power Analysis - Pre-Experiment Planning:**

**Critical Importance:**
"Before running an experiment, knowing how long it should run and how many participants you need to obtain reliable results/desired MDE is crucial."

**Purpose:**
- Helps not waste cycles on impossible experiments
- Sets correct expectations for experiment durations
- Determines sample size required to detect effect of certain size with given confidence
- Ensures experiments neither underpowered (risk missing true effect) nor overpowered (wasting resources)

**Power Analysis Process:**

1. **Select Population:**
   - Based on past experiment, qualifying event, or entire user base
   - Choose appropriate population for experiment

2. **Choose Metrics:**
   - Input metrics for evaluation criteria
   - Can add multiple metrics to analyze sensitivity

3. **Run Simulation:**
   - Tool simulates experiment
   - Calculates population sizes and variance based on historical behavior
   - Shows week-by-week simulation results
   - Estimates number of users eligible each day from historical data

4. **Review and Adjust:**
   - If estimated sample sizes not achievable: adjust MDE or extend duration
   - Balance between detection sensitivity and feasibility

**The Patience Principle:**

**Key Rule of Thumb:**
"Patience leads to confidence."

**Why Patience Matters:**
- Confidence is KEY when examining test results
- Early results can suggest clear path but may be misleading
- Temptation to make swift decisions understandable but dangerous
- Allure: faster iteration cycle, quick validation, ability to pivot/ship quickly
- Reality: premature decisions based on insufficient data

**Three Techniques for Safe Early Decision-Making:**

**1. Sequential Testing:**

**Purpose:**
- Enable early decision-making when sufficient evidence
- Limit risk of false positives
- Identify unexpected regressions early
- Distinguish significant effects from random fluctuations

**How It Works:**
- Evaluate experiment results at MULTIPLE points during experiment
- Does NOT increase risk of false positives (unlike naive peeking)
- Applies adjustment to p-values and confidence intervals
- Accounts for increased false positive rate associated with peeking

**When to Use:**
- Significant opportunity cost to delaying experiment decision
- Need to catch regressions quickly
- Improvement in key metrics observed early

**Caution:**
- Early stat-sig results for certain metrics don't guarantee sufficient power to detect regressions in OTHER metrics
- Still need to consider multiple metrics and overall experiment health

**2. CUPED (Controlled-Experiment Using Pre-Experiment Data):**

**Benefits for Early Detection:**
- Shrinks confidence intervals and p-values
- Statistically significant results achievable with SMALLER sample size
- REDUCES duration required to run experiment
- Less data needed to reach conclusive results

**Effectiveness Conditions:**
- Most effective when pre-experiment behavior CORRELATED with post-experiment behavior
- Not suitable for new users without historical data
- Requires stable behavioral patterns

**Empirical Results:**
- Based on 300+ customers utilizing CUPED within Statsig
- Can reduce required sample sizes significantly
- Reduces experiment run times due to lower metric variance
- Variance reduction translates directly to faster detection

**3. Multi-Armed Bandits (MAB):**

**Core Mechanism:**
- Dynamically adjusts traffic allocation towards different variations based on performance
- Unlike traditional A/B: traffic NOT split evenly, allocation CHANGES during test
- Continuously shifts more users to better-performing treatments

**Advantages:**
- Maximizes potential reward DURING experiment (not just at end)
- Reduces opportunity cost of showing sub-optimal variations
- Minimizes risk by progressively allocating less traffic to underperforming treatments
- Useful for continuous experimentation without fixed end date

**Limitations:**
- Typically optimizes for SINGLE metric only
- Not ideal when multiple metrics matter equally
- Consideration needed when planning experiments

**Decision Framework Summary:**

**Appropriate for Early Decisions (24-48 hours):**
- Catastrophic regressions (major bugs, crashes)
- Severe negative impacts on key metrics
- Technical failures requiring immediate action
- Clear, overwhelming evidence with proper statistical adjustments

**NOT Appropriate for Early Decisions:**
- Subtle effect sizes requiring full power
- Multiple metrics needing evaluation
- Situations where novelty effects likely
- Ship/no-ship decisions on marginal improvements
- Precise effect size estimation needed

**Best Practice:**
"While early experiment results can be informative, they should always be taken with a grain of salt. A disciplined approach to experimentation, supported by robust tools, is essential for making informed decisions."

**Key Principle - Distinguish Learning from Decision-Making:**
- Early signals (24-48 hours) = LEARNING signals
  - Technical diagnostics
  - Randomization validation
  - Catastrophic issue detection
  - Early trend observation (with caveats)

- Later signals (7+ days) = DECISION signals
  - Ship/no-ship decisions
  - Effect size estimation
  - Multi-metric evaluation
  - Long-term impact assessment

## Phase 4 (continued): Guardrail Metrics

### Source: Statsig - What are guardrail metrics in A/B tests?
URL: https://www.statsig.com/blog/what-are-guardrail-metrics-in-ab-tests

**The Hidden Trap Door Scenario:**
- Launch feature, primary metric improves MORE than goal
- Celebrate success
- Three months later: costs have TRIPLED
- Unintended consequences are universal constant
- Even when primary metrics look great, not everything may be peachy

**Core Definition:**
"Like a guardrail that keeps your car on the road, they are your safety net."

**Purpose:**
- Monitor critical aspects of product that could be negatively affected by changes tested
- Ensure gains in one area don't cause losses in another
- Provide holistic view of impact of tests
- Guard against potential negative side effects
- Preserve user experience and system functionality
- Save resources and shield product from unintended consequences

**Primary Metrics vs. Guardrail Metrics:**

**Primary Metrics:**
- Target outcomes directly tied to experiment objectives
- Things you're actively trying to MOVE
- Example: click-through rate on feature button
- Focus of optimization effort

**Guardrail Metrics:**
- Safeguard broader health of product
- Monitor while focusing on primary goals
- Example: page load times, user error rates
- Ensure improvements in one area don't degrade overall satisfaction/functionality

**Together:**
- Form balanced strategy
- Optimize performance while maintaining quality user experience
- Ensure A/B tests improve specific features without compromising system integrity
- Enable informed, impactful decisions benefiting both product and users

**Not Just for Mistakes:**

**Critical Understanding:**
"Guardrail metrics are not just for finding errors. A degradation in one does not mean you made a mistake."

**Degradations Can Be Due To:**
- Complex systems
- Downstream effects
- Interactions with other features
- Delayed effects

**Proper Framing:**
- Insurance policy, NOT blame game
- Tool to find issues faster
- Ideally find simple fix or rethink product for even better outcomes

**Real-World Examples:**

**Airbnb:**
- Balance user experience with business growth
- Test changes to increase bookings
- Monitor guest satisfaction scores closely
- Ensure boosting one metric doesn't adversely affect overall customer satisfaction

**Netflix:**
- Monitor stream start times and buffering ratios
- Test algorithms for personalized recommendations
- Ensure core user experience (smooth streaming) remains unaffected
- Prevent potential subscriber drop-off due to performance issues

**Uber:**
- Test new matching algorithm to improve trip conversion rate
- Guardrail: ensure success isn't simply from reducing overall rider requests
- Prevent false positives from unintended mechanism

**Selecting Effective Guardrail Metrics:**

**No One-Size-Fits-All:**
- Rely on knowledge of: product, users, business, data
- Make informed decision about what to measure

**Strategy 1 - Core Business Objectives:**
- Choose guardrails from core business objectives
- Metrics reflecting health of product/service
- Example: if customer retention key → monitor user churn, session duration
- Apply regardless of what's changing in A/B test

**Strategy 2 - Anticipate What Could Go Wrong:**
- Identify potential drop-offs in funnel
- If one degraded, could it make primary metrics accidentally look better?
- Double-check results really are as good as they look
- Prevent false positives from unintended mechanisms

**Strategy 3 - Technical Issues:**
- Degraded page load time
- Increase in user retries
- Increased bug report rate
- Classic technical health indicators

**When to Be Restrained:**

**1. Risk of Noise:**
- Every metric subject to real-world noise
- Random chance WILL result in some "statsig" detections that are just noise
- At 5% significance (95% confidence): expect 5% of metric results to be "statsig" different even if no real difference exists
- More metrics = more chances any single metric is noisy
- Too many metrics can cloud decision-making
- Ensure guardrail metrics provide clear insights without data overload

**2. Sensitivity and Statistical Power Limitations:**
- No guarantee experiment has sensitivity to measure deltas in every guardrail metric
- Experiment sample size likely designed based on PRIMARY metric
- Primary metric variance might look entirely different than guardrail's
- May be set up to measure primary metric MDEs of 1%
- But business-critical guardrail only statistically detectable at 10%, 20%, even 30% change
- Reflected in huge confidence intervals around metric

**Best Practice:**
- Choose guardrails meaningful and well-powered at expected sample sizes
- Run secondary power analyses (after primary) to check if guardrails meaningful for given design sample size

**3. Defining Meaningful Thresholds:**
- Have prior idea of "how bad is bad enough" to be concerned
- Example: 1% slowdown in page load times
  - Concerning enough to roll back?
  - Do benefits outweigh slowdown?
  - Is 1% huge concern?
- Lean on knowledge of particular business and product
- Make decisions for each launch based on context

**Implementation on Statsig:**
- Guardrail metrics are just normal metrics
- Add to "Secondary Metrics" section in experiment setup

**Monitoring Approaches:**

**Option 1: Bug Detection (Classic Approach):**
- Classic A/B test designed to be checked and analyzed ONCE
- Shouldn't end experiments early
- Best practice: avoid "peeking" at metrics in general
- Avoid temptation to declare success due to early, noisy data

**Exception for Guardrails:**
- Can check guardrail metrics for obvious bugs
- Ensure they haven't tanked
- Identify issues sooner that entirely break experience/user flow
- Outright-breaking changes should be easy to see
- Deltas should be large and obvious

**Pro Tip:**
- Use automated alerts with reasonable thresholds
- Represent obvious regressions that should stop test
- Avoid temptation to look at other metrics and make early decisions
- Beyond checking for bugs: avoid looking at primary metrics
- Ignore promising results, trust design, let experiment run through design duration

**Option 2: Continuous Monitoring (Sequential Testing):**
- Use Sequential Testing to allow continuous monitoring
- Can look at results as much as wanted
- Still avoid increasing chances of false discovery
- Keep close eye ensures significant changes don't go unnoticed
- Proactive approach maintains integrity of product experience and business health
- Regularly check dashboard and adjust strategies as needed

**Implications for 24-48 Hour Diagnostics:**

**Highly Reliable Early Signals:**
- Catastrophic guardrail violations (crashes, errors)
- Technical health metrics (load times, error rates)
- Obvious breaking changes
- Large, unmistakable degradations

**Less Reliable Early Signals:**
- Subtle guardrail movements
- Metrics requiring high statistical power
- Metrics with high variance
- Delayed effects not yet manifested

**Key Principle:**
"Guardrails are powerful, but more is not always better. And sometimes your desired guardrail will simply never be useful."

**Critical for 24-48 Hours:**
- Guardrail metrics among MOST reliable early signals
- Technical issues manifest quickly
- Breaking changes immediately visible
- Can safely monitor without waiting for full experiment duration
- Exception to "don't peek" rule for good reason

## Phase 4 (continued): Leading vs. Lagging Indicators and Proxy Metrics

### Sources: Multiple - Amplitude, Netflix Tech Blog, Product Management Literature

**Core Definitions:**

**Leading Indicators:**
- Help predict FUTURE performance
- Measurable today with confidence we're heading in right direction
- Early warning signals of impact on outcomes
- Can predict what is likely to happen
- Measure momentum before final results show
- Early signals measured quickly that reliably predict outcomes you care about

**Lagging Indicators:**
- Give insight into PAST outcomes
- Measure outcomes and tell what already happened
- Confirm long-term results
- Examples: revenue, retention, lifetime value

**Proxy Metrics:**
- Specific type of leading indicator
- Early indicators of future outcomes
- Measurable indicators that predict movement in more important but slower-to-capture outcomes
- Biomarkers or intermediate outcomes used as substitutes for clinical outcomes of interest
- Used to expedite research or decision-making

**Why They Matter in Experiments:**

**Speed Advantage:**
- Leading indicators provide faster feedback
- Don't have to wait for long-term outcomes to manifest
- Enable quicker iteration and decision-making
- Particularly valuable in first 24-48 hours

**Prediction Capability:**
- Good proxy metrics reliably predict downstream outcomes
- Can measure quickly but predict retention, LTV, etc.
- Netflix research: learn good proxy metrics from historical experiments
- Some proxies can be 8x more sensitive than final outcomes

**Risk:**
- Proxy metrics must be validated against actual outcomes
- Poor proxies can mislead decision-making
- Need historical data to establish correlation
- Surrogate endpoints require careful validation

**Examples in Product Context:**

**Leading Indicators (Early Signals):**
- Activation events (first key action taken)
- Early engagement (sessions in first week)
- Feature adoption rate
- Time to first value
- Onboarding completion
- Click-through rates
- Page views

**Lagging Indicators (Decision Metrics):**
- Revenue
- Customer retention (30/60/90 day)
- Lifetime value
- Churn rate
- Net Promoter Score
- Customer satisfaction

**Validation Requirements:**

**For Proxy Metrics to Be Reliable:**
1. Strong correlation with ultimate outcome
2. Validated through historical data
3. Measured consistently
4. Causally related (not just correlated)
5. Sensitive enough to detect meaningful changes

**Netflix Approach:**
- Learn proxy metrics from past experiments
- Validate which early signals predict long-term outcomes
- Use historical experiment data to identify best proxies
- Continuously refine proxy metric selection

**Implications for 24-48 Hour Diagnostics:**

**What Can Be Measured Early (Leading Indicators):**
- Technical performance metrics
- Immediate user actions (clicks, views)
- Early engagement signals
- Activation events
- Error rates
- Load times

**What Cannot Be Measured Early (Lagging Indicators):**
- Long-term retention
- Lifetime value
- Revenue impact (unless very high volume)
- Customer satisfaction (needs time to form)
- Churn (by definition requires time)

**Critical Distinction for Decision-Making:**
- Use leading indicators for LEARNING in first 24-48 hours
- Use lagging indicators for DECISIONS after sufficient time
- Validate that leading indicators actually predict lagging indicators
- Don't confuse early movement in proxy with guaranteed long-term impact

**Key Principle:**
"Leading indicators are a way of measuring things today with a level of confidence that we're heading in the right direction."

But confidence ≠ certainty. Early signals must be:
- Interpreted with caution
- Validated against historical patterns
- Combined with other diagnostics
- Distinguished from decision metrics
